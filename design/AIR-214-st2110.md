# AIR-214 — SMPTE ST 2110 support (epic): stack survey + phased plan

> Researched 2026-07-16 (web survey, primary sources verified). Decision doc
> for the AIR-214 epic; implementation tickets are cut from the phases below.

## The shape of the problem

- **Receive and transmit are radically asymmetric.** 2110-20 receive is a
  throughput/buffering problem — proven on tuned kernel sockets at 3 Gb/s
  zero-loss since 2017 (EBU/FFmpeg). **Compliant *narrow* (2110-21) transmit is
  a sub-microsecond pacing problem** (~7 µs inter-packet at HD): every credible
  sender offloads pacing to NIC hardware. EBU: narrow is "probably impossible"
  in pure software. The standard's "wide" sender type exists for software but
  shallow-buffer FPGA receivers may reject it.
- **Bandwidth reality:** 1080p59 2110-20 is **~2.4–2.6 Gb/s and >200k pkt/s per
  flow** — 25 GbE is the normal endpoint attachment; 10 GbE fits ~3 HD flows.
- **PTP (ST 2059-2)** required in practice: linuxptp 3.1.1 (Ubuntu 22.04) with a
  hand-written SMPTE profile (domain 127, logSync −3, E2E) + NIC hardware
  timestamps (`ethtool -T`). Receive-only can run unlocked (type-A receiver);
  transmit cannot.
- **TR-1001-1 plants also expect** ST 2022-7 dual-network protection and NMOS
  IS-04/IS-05 (procurement-mandatory) — without NMOS, Airlock is manual-SDP-only.
- **Peer precedent:** vMix/OBS have no native 2110; high-end software (Telestream,
  GV AMPP, TAG) rides NVIDIA Rivermax. Nobody at our tier does it natively.

## Phased plan

- **Phase 0 (now, docs only):** customers needing Airlock in a 2110 plant use an
  edge gateway or a DeckLink IP card in a bridge PC; PTP/NMOS handled in-box.
- **Phase 1 — receive-only -20/-30/-40:** a new child process feeding the
  existing shm rings (the SRT/SCTE pattern). GStreamer 1.28+ pipeline
  (`udpsrc → rtpvrawdepay / rtpL24depay / rtpsmpte291depay`, RFC 7273 clock)
  behind a seam that can also host the MTL child (kernel/AF_XDP mode) — same
  code path unlocks DPDK/E810 later. linuxptp + phc2sys host config; HW
  timestamping pre-flight. Scope guards: progressive-only, single interface
  (TR-1001-1-legal), manual SDP.
- **Phase 2 — NMOS IS-04/IS-05 node:** NVIDIA NvNmos (single C library wrapping
  sony/nmos-cpp, SDP strings in/out — friendliest P/Invoke) with IS-05 wired
  into the channel model. Own epic-sized task; mind SettingsSyncMask and
  SuppressExternalOutputs on backups.
- **Phase 3 — transmit:** wide sender (declared `TP=2110TPW`) via MTL on any
  NIC, or 2110-22 JPEG XS (Vectis patent pool — royalties); **narrow only when
  a plant demands it**: MTL + Intel E810/E830 + DPDK (free, third-party
  verified) or Rivermax + ConnectX (paid). ST 2022-7 dual-stream lands here.

## Risks

GStreamer 2110 elements are young (pin plugins-rs ≥0.15.x; keep the MTL child
as plan B) · interlaced receive forces the FFmpeg/MTL path · 2110-31 (Dolby E)
exists only in Rivermax or a gateway · narrow-TX couples fleet procurement to
E810/ConnectX and brings DPDK ops (VFIO ports, hugepages) · treat 2110 as
Linux-first (Windows parity is weakest everywhere).
